# Refactor Plan for Activation Module

## 1. Introduction

This document outlines a plan to refactor the application's activation module to improve code quality, enhance error handling, and seamlessly integrate with the new `license.json` file format generated by the `smc-key-gen-v1` CLI.

The core goal is to make the activation process more robust, secure, and user-friendly by clearly separating the UI from the business logic.

## 2. Current Problems

Based on the existing files (`validator.ts`, `check-activation-key.ts`, `activate-key.ts`, `activate-key.tsx`), the following issues have been identified:

- **Tight Coupling:** Activation logic is spread across multiple files and is tightly coupled with the `Setting` database model, making it hard to test and maintain.
- **Unstructured Error Handling:** The code throws simple string errors (`alert("ไม่สามารถ activate ได้")`), making it difficult for the UI to handle different error types gracefully and provide specific feedback to the user.
- **Lack of Centralized Logic:** The activation process is a series of isolated function calls, not a cohesive workflow. This leads to redundant logic and complex state management in the UI.
- **Outdated Key Format:** The existing code handles a plain text license key, not the new JWT-based `license.json` file.

## 3. Refactoring Goals

The refactoring aims to achieve the following:

- **Seamless Integration:** The application must be able to read and validate the new `license.json` file format.
- **Robust Error Handling:** Implement a standardized error handling mechanism using a custom error class. The UI should display specific and helpful error messages based on the error type (e.g., "Invalid Key Format", "License Expired", "Hardware Mismatch").
- **Clear Separation of Concerns:** Create a dedicated `ActivationService` to encapsulate all business logic, making the UI component's responsibility limited to handling user interaction and displaying status.
- **Improved Code Structure:** Centralize the activation workflow, reducing redundancy and improving readability.
- **Security:** Ensure the JWT validation process is secure by using a hard-coded, obfuscated `SECRET_KEY` as planned in the CLI application.

## 4. Refactoring Approach (Atomic Tasks)

### Step 1: Create Custom Error Handling Classes

- **Task 1.1:** Create a new file `src/errors/activationError.ts`.
- **Task 1.2:** Define an `enum` named `ActivationErrorCode` to represent specific error types.
  - `INVALID_FILE_FORMAT`
  - `INVALID_SIGNATURE`
  - `LICENSE_EXPIRED`
  - `HARDWARE_MISMATCH`
  - `UNEXPECTED_ERROR`
- **Task 1.3:** Define a custom `ActivationError` class that extends `Error` and includes the `ActivationErrorCode`.

- **Success Criteria:** The new `ActivationError` class is defined and the `ActivationErrorCode` enum contains all necessary error types.

### Step 2: Create a Central `ActivationService`

- **Task 2.1:** Create a new file `src/services/activationService.ts`.
- **Task 2.2:** Define a public asynchronous function `activateFromLicenseFile(filePath: string): Promise<void>`. This will be the main entry point for the activation logic.

- **Success Criteria:** The `ActivationService` file exists and contains the main `activateFromLicenseFile` function, which serves as the single entry point for the activation logic.

### Step 3: Implement Core Activation Logic within `ActivationService`

- **Task 3.1: Read the license file.**
  - Use `fs.readFileSync` to read the file content from the provided `filePath`. Add a `try...catch` block to handle file system errors.
  - **Success Criteria:** The function successfully reads the file and handles file system errors by throwing a specific `ActivationError`.
- **Task 3.2: Parse the JSON and extract the JWT.**
  - Use `JSON.parse` to parse the file content.
  - Add a check to ensure the JSON contains a `token` property.
  - **Success Criteria:** The function correctly parses the JSON and checks for the `token` property. It throws `ActivationError(ActivationErrorCode.INVALID_FILE_FORMAT)` if the file format is incorrect.
- **Task 3.3: Verify and Decode the JWT.**
  - Import the obfuscated `SECRET_KEY`.
  - Use `jwt.verify(token, SECRET_KEY)` to verify the signature and decode the payload.
  - **Success Criteria:** The function successfully verifies the JWT signature and throws `ActivationError` with codes `LICENSE_EXPIRED` or `INVALID_SIGNATURE` if the validation fails.
- **Task 3.4: Perform Hardware Validation.**
  - Get the current machine's `Hardware ID` using the existing `getHardwareId()` function.
  - Compare it with the `mac_address` inside the decoded JWT payload.
  - **Success Criteria:** The function must obtain the hardware ID and compare it with the `mac_address` in the JWT payload. It must throw `ActivationError(ActivationErrorCode.HARDWARE_MISMATCH)` if they do not match.
- **Task 3.5: Save the new license key.**
  - Call a new internal function `_saveLicense(token)` to save the validated JWT token to the database.
  - The `_saveLicense` function should handle all database interactions.
  - **Success Criteria:** The `_saveLicense(token)` function successfully saves the JWT to the database via the `Setting` model.
- **Task 3.6: Refactor `validator.ts`.**
  - **Success Criteria:** The old `validateLicense` and `activateLicense` functions are removed from `validator.ts`. Only the `getHardwareId` and other reusable helpers remain.

### Step 4: Refactor UI Component `activate-key.tsx`

- **Task 4.1: Update file input.**
  - Change the UI to use a file input (`<input type="file" />`) instead of a text input.
  - **Success Criteria:** The component renders a file input element, and its `onChange` handler correctly captures the selected file.
- **Task 4.2: Manage UI state.**
  - Use a single state object to manage the UI status (e.g., `status: 'idle' | 'activating' | 'success' | 'error'`).
  - **Success Criteria:** The UI component's state is managed by a single object, simplifying rendering logic.
- **Task 4.3: Implement the activation handler.**
  - The `onSubmit` handler will now get the selected file from the `event` and call `activateFromLicenseFile(filePath)` from the `ActivationService`.
  - Use a `try...catch` block to handle `ActivationError`.
  - **Success Criteria:** The UI handler successfully calls the service and updates its state and error messages based on the result from the `ActivationService`.
